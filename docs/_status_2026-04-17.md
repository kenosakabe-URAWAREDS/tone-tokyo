## 3. /editor 認証方式

- **実装:** 自前Cookie認証（`lib/editor-auth.ts` + `proxy.ts` + `app/api/editor/login/route.ts`）
- **パスワード管理:** `EDITOR_PASSWORD` 環境変数で管理。`.env.local` に存在、ハードコードなし ✅
- **セッショントークン:** `createHmac('sha256', password).update('tone-tokyo-editor-v1').digest('hex')` を Cookie `tone_editor_auth` に保存。
  - **Cookie にパスワード平文は入らない**設計（HMACの固定文字列版を格納）→ Cookie が盗まれてもパスワードは露出しない
  - **ただし** Cookie が盗まれればセッションごと乗っ取られる（通常のセッション Cookie と同等のリスク）
- **timing-safe 比較:** `timingSafeEqual` 使用 ✅
- **Cookie options:** `httpOnly`, `sameSite: 'lax'`, `secure`（本番のみ）, `maxAge: 24h` ✅
- **ゲートの仕組み:** `proxy.ts` が `/editor/:path*` と `/api/editor/:path*` に matcher。`/editor` 自体は公開（ログインフォーム描画のため）。未認証で API にアクセス → 401 JSON。未認証で下層ページ → `/editor?next=...` へ 302 redirect。
- **セキュリティ的な懸念点:**
  1. **単一パスワード・単一ユーザー**: 将来 Felix やアシスタント編集者を増やす場合、個別アカウントが必要になる可能性。現状は KAKEHASHI 運営・刑部さんのみを想定。
  2. **ローテーション手順なし**: パスワードを変更するとアクティブな Cookie は maxAge 切れまで無効化されない（HMAC key が変わるため、新パスワードでの Cookie 値と旧 Cookie 値が一致しなくなり実質無効化される→問題なし）。
  3. **ログ/監査なし**: ログイン成功・失敗のロギングがない。
  4. **レート制限なし**: `/api/editor/login` にブルートフォース対策なし。`EDITOR_PASSWORD` が十分長く複雑であれば実用上の問題は小さいが、Phase 1 開始前に検討する価値はある。
  5. **全 `/api/editor/*` が一律保護**だが、`sanity-token` エンドポイントのように Sanity Write Token を返すものがあり、認証突破されると Sanity 書込が可能になる点は構造上のトレードオフ。現状の単一パスワード設計では妥当。

**次パートへの申し送り:** 指示書パート4（/editor 統合管理画面化）で認証方式を差し替える必要があるかどうかの判断が必要。**現状は安全に運用可能な設計**と判断。パスワード変更以外の抜本的な刷新は Phase 1 後でよいと推奨。

## 4. Sanity スキーマ現状（article 型）

公開スキーマは `sanity/schemas/index.ts` 一本に集約。主要フィールド:

### 基本情報
- `title`（required, string）
- `titleJa`（string）
- `slug`（required, slug, title から自動）
- `pillar`（required, `FASHION | EAT | CULTURE | EXPERIENCE | CRAFT | FAMILY`）※ 指示書で「5ピラー」と記載だが、**スキーマには `FAMILY` も含まれている**
- `subtitle`（text）
- `heroImage`（image + hotspot）/ `heroImageUrl`（url）/ `heroCaption`（string）
- `gallery`（image array）
- `body`（PortableText array: block + image）
- `bodyJa`（plain text — 日本語レビュー用）

### 共通フィルタ
- `area`（Tokyo / Osaka / Kyoto / Fukuoka / Okayama / Kurume / Hokkaido / Okinawa / Nagoya / Kobe / Other）
- `neighborhood`（string）
- `editorRating`（1|2|3 = Worth a Visit / Highly Recommended / Must Go）

### ピラー固有フィルタ（`hidden` 条件で pillar ごとに分岐）
- EAT: `eatGenre`, `bookingDifficulty`, `drinks`, `scene`, `eatPriceRange`, `priceRange`（高級店用）
- FASHION: `fashionCategory`, `fashionType`, `articleType`, `fashionPriceRange`, `fashionFeature`
- CULTURE: `cultureCategory`, `cultureVibe`, `cultureTime`
- EXPERIENCE: `experienceCategory`, `experienceFor`, `experienceDuration`, `experienceSeason`
- CRAFT: `craftCategory`, `craftType`, `craftVisit`

### 場所情報
- `locationName`, `locationNameJa`, `officialUrl`, `googleMapsUrl`, `tabelogUrl`, `address`, `phone`, `hours`, `websiteUrl`, `referenceUrls`

### Japanese Abroad シリーズ
- `isJapaneseAbroad`（boolean, default false）, `city`, `country`

### メタ
- `tags`（array, layout: tags）
- `readTime`（string）
- `status`（`draft | review | published | scheduled`, default draft。**未設定は「公開扱い」に運用**）
- `publishedAt` / `scheduledAt`（datetime）
- `seoTitle`, `seoDescription`
- `sourceType`（`kentaro-initiated | ai-curated`）

### パート3（スキーマ改修）で不足しているフィールド（指示書方針から推定）
- `articleType`（`editorial | news`）← Editorial/News 2区分
- `disclosure`（boolean or reference to brand）← 4段階ブランド言及ルール
- `brandInvolved`（string or reference）← Blues Inc. 系ブランド等の開示対象
- `faq`（array of {question, answer}）← JSON-LD FAQPage 用の構造化データ
- `answerBlock`（text）← AEO 向けの冒頭アンサー
- その他 AEO/AI臭さスコア関連フィールド

## 7. LINE Bot 現状（`app/api/line-webhook/route.ts`）

- **現状:** capture-only モード。
  1. LINEから text / photo を受信
  2. `stockpile`（ネタ帳）ドキュメントとして Sanity に保存
  3. 「📥 ネタ帳に保存しました」と一度だけ返信して終了
- **旧実装の教訓:** webhook内で Claude 呼び出しするとタイムアウトで「記事を生成中」のまま固まるため、生成は `/editor` から手動トリガーする設計に変更済み。
- **バッチング:** 1つの webhook POST に複数イベント（text + photo）が混ざる場合、同一 userId でまとめて1つの stockpile にする（キャプション付き写真を分散保存しない工夫）。
- **Google Maps / Tabelog URL 抽出:** 正規表現で text から剥がし、stockpile の専用フィールドへ。
- **音声入力:** 未対応（指示書の予想通り）。LINE Messaging API の audio message type をハンドルしていない。Phase 1 の「音声メモ5分/日」導線を作るならパート5で追加実装が必要。
